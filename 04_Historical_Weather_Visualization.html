Part 1: Getting Historical Weather Data

The OpenWeather One Call API 3.0 provides historical daily summaries for a given latitude, longitude, and date. This requires coordinates — which is why we built the geocode() function!

The Plan

  1. Use the Geocoding API to get lat/lon for a city
  2. Define a range of past dates
  3. Loop over each date, calling the One Call API
  4. Combine all results into a single data frame

NoteWhy as.character()?

Without wrapping in as.character(), dates will mathematically evaluate and become integers when used in URL construction. We need them as strings like "2026-08-05" for the API.

# Step 4: Loop over dates and call the API
hist_weather_df <- data.frame()

for (date in date_range) {
  
  weather_url <- glue(
    "https://api.openweathermap.org/data/3.0/onecall/day_summary?",
    "lat=", lat,
    "&lon=", lon,
    "&date=", date,
    "&appid=", Sys.getenv("API_KEY"),
    "&units=imperial"
  )
  
  weather_response <- req_perform(request(weather_url))
  
  if (resp_status(weather_response) == 200) {
    daily_weather_df <- resp_body_json(weather_response) |>
      as.data.frame()
    
    # Add city and date columns
    daily_weather_df <- daily_weather_df |>
      mutate(
        city = city_name,
        date = date
      )
    
    hist_weather_df <- bind_rows(hist_weather_df, daily_weather_df)
    
  } else {
    warning(paste("Failed to get weather for", date, "-", resp_status(weather_response)))
  }
}

print(hist_weather_df)

Understanding the Response Structure

The API returns a JSON object representing summary statistics for a specific location and date. The nested structure flattens into descriptive column names:

JSON Key R Column Name Type Description
temperature.min temperature.min numeric Minimum temp for the day
temperature.max temperature.max numeric Maximum temp for the day
wind.max.speed wind.max.speed numeric Peak wind speed
date (added) date character From our loop variable
city (added) city character City name from geocoding

Example row:

temperature.min temperature.max wind.max.speed date city
50.1 75.6 12.3 2026-08-04 San Luis Obispo

Q1.

(Open-Ended): In the loop above, we use bind_rows() to combine data from each day. Why do we initialize hist_weather_df <- data.frame() before the loop? What would happen if we didn’t?

TipAnswer Q1

We initialize an empty data frame before the loop so that bind_rows() has something to append to on the first iteration. Without it, R would throw an error because hist_weather_df wouldn’t exist when we try to bind the first day’s results to it. This is a common pattern in R: initialize → loop → accumulate.

Part 2: Building a Reusable Function

Let’s wrap the logic above into a clean, reusable function:

prev_weather <- function(city, days) {
  
  # --- Geocoding ---
  geo_url <- glue(
    "http://api.openweathermap.org/geo/1.0/direct?",
    "q=", URLencode(city),
    "&limit=1&appid=", Sys.getenv("API_KEY")
  )
  
  geo_response <- req_perform(request(geo_url))
  
  if (resp_status(geo_response) != 200) {
    stop("Geocoding failed. Check your API key or city name.")
  }
  
  geo_data_df <- resp_body_json(geo_response) |> as.data.frame()
  
  if (length(geo_data_df) == 0) {
    stop("City not found. Please check the city name.")
  }
  
  lat <- round(geo_data_df$lat, digits = 2)
  lon <- round(geo_data_df$lon, digits = 2)
  cat("Coordinates for", city, "-> Lat:", lat, "Lon:", lon, "\n")
  
  # --- Historical Weather ---
  date_range <- as.character(lubridate::today() - days(1:days))
  hist_weather_df <- data.frame()
  
  for (date in date_range) {
    weather_url <- glue(
      "https://api.openweathermap.org/data/3.0/onecall/day_summary?",
      "lat=", lat,
      "&lon=", lon,
      "&date=", date,
      "&appid=", Sys.getenv("API_KEY"),
      "&units=imperial"
    )
    
    weather_response <- req_perform(request(weather_url))
    
    if (resp_status(weather_response) == 200) {
      daily_weather_df <- resp_body_json(weather_response) |>
        as.data.frame() |>
        mutate(city = city, date = date)
      
      hist_weather_df <- bind_rows(hist_weather_df, daily_weather_df)
    } else {
      warning(paste("Failed to get weather for", date, "-", resp_status(weather_response)))
    }
  }
  
  return(hist_weather_df)
}

TipAnswer Q3

Each iteration constructs a new URL (with a different date), creates a request, performs it, checks the status, and parses the result. The added complexity is the accumulation step — binding each day’s data frame into a growing combined result.

library(sf)
library(rnaturalearth)
library(rnaturalearthdata)
cities <- c("San Luis Obispo", "Santa Barbara", "San Francisco")

# Function to get coordinates + weather summary for a single city
get_city_weather <- function(city, date = Sys.Date()) {
  geo_url <- glue(
    "http://api.openweathermap.org/geo/1.0/direct?",
    "q=", URLencode(city),
    "&limit=1&appid=", Sys.getenv("API_KEY")
  )
  
  geo_response <- req_perform(request(geo_url))
  
  if (resp_status(geo_response) == 200) {
    geo_data_df <- as.data.frame(resp_body_json(geo_response))
    
    if (length(geo_data_df) == 0) return(NULL)
    
    lat <- geo_data_df$lat
    lon <- geo_data_df$lon
    
    weather_url <- glue(
      "https://api.openweathermap.org/data/3.0/onecall/day_summary?",
      "lat=", lat,
      "&lon=", lon,
      "&date=", format(date, "%Y-%m-%d"),
      "&appid=", Sys.getenv("API_KEY"),
      "&units=imperial"
    )
    
    weather_response <- req_perform(request(weather_url))
    
    if (resp_status(weather_response) == 200) {
      weather_data <- resp_body_json(weather_response)
      
      tibble(
        city = city,
        date = date,
        lat = lat,
        lon = lon,
        temp_max = weather_data$temperature$max,
        temp_min = weather_data$temperature$min
      )
    } else return(NULL)
  } else return(NULL)
}

# Fetch for all cities
weather_df <- bind_rows(lapply(cities, get_city_weather))
# Convert to spatial data
weather_sf <- weather_df |>
  st_as_sf(coords = c("lon", "lat"), crs = 4326)

# Get California boundary
ca <- ne_states(country = "United States of America", returnclass = "sf") |>
  filter(name == "California")

# Plot
ggplot() +
  geom_sf(data = ca, fill = "gray95", color = "gray60") +
  geom_sf(data = weather_sf, aes(color = temp_max), size = 6) +
  geom_sf_label(data = weather_sf, aes(label = city), nudge_y = 0.5, size = 2.5) +
  scale_color_viridis_c(option = "plasma", name = "High Temp (°F)") +
  labs(
    title = "California City Temperatures",

Q4.

(Open-Ended): We used lapply(cities, get_city_weather) to apply the function to multiple cities. How is this different from using a for loop? When might you prefer one over the other?

TipAnswer Q4

lapply() applies a function to each element of a list and returns a list of results — it’s a more “functional programming” approach. A for loop does the same thing but requires manually initializing and accumulating results. lapply() is more concise and idiomatic in R. However, for loops can be easier to read and debug, especially when you need to handle errors or track progress. For API work with potential failures, a for loop with tryCatch or warning() can be more transparent.

Part 5: Reflection & Pitfalls

Common Pitfalls When Working with APIs

  1. Rate Limits: Most APIs limit how many requests you can make per minute/hour. The free OpenWeather tier allows 60 calls/minute. Design your code to respect these limits.

  2. Authentication Expiry: API keys can expire or be revoked. Always check your key status if you get 401 errors.

  3. Data Structure Changes: API providers can change their response format without warning. Always inspect the structure of new responses before building pipelines around them.

  4. URL Encoding: Special characters (spaces, ampersands, accents) must be encoded. Use URLencode().

  5. Nested JSON Complexity: Real-world JSON can be deeply nested. Practice with str() and names() to navigate the structure before trying to flatten it.

Reflection Questions for Your Practice

  • What are the pitfalls (rate limits, authentication) that students need to know?
  • How does working with live, messy data differ from working with pre-cleaned datasets?
  • What ethical considerations arise when collecting data from APIs?

# Function to get coordinates + weather summary for a single city
get_city_weather <- function(city, date = Sys.Date()) {
  geo_url <- glue(
    "http://api.openweathermap.org/geo/1.0/direct?",
    "q=", URLencode(city),
    "&limit=1&appid=", Sys.getenv("API_KEY")
  )
  
  geo_response <- req_perform(request(geo_url))
  
  if (resp_status(geo_response) == 200) {
    geo_data_df <- as.data.frame(resp_body_json(geo_response))
    
    if (length(geo_data_df) == 0) return(NULL)
    
    lat <- geo_data_df$lat
    lon <- geo_data_df$lon
    
    weather_url <- glue(
      "https://api.openweathermap.org/data/3.0/onecall/day_summary?",
      "lat=", lat,
      "&lon=", lon,
      "&date=", format(date, "%Y-%m-%d"),
      "&appid=", Sys.getenv("API_KEY"),
      "&units=imperial"
    )
    
    weather_response <- req_perform(request(weather_url))
    
    if (resp_status(weather_response) == 200) {
      weather_data <- resp_body_json(weather_response)
      
      tibble(
        city = city,
        date = date,
        lat = lat,
        lon = lon,
        temp_max = weather_data$temperature$max,
        temp_min = weather_data$temperature$min
      )
    } else return(NULL)
  } else return(NULL)
}

### Q4. {.unnumbered}

**(Open-Ended):** We used `lapply(cities, get_city_weather)` to apply the function to multiple cities. How is this different from using a `for` loop? When might you prefer one over the other?

::: {.callout-tip collapse="true" title="Answer Q4"}
`lapply()` applies a function to each element of a list and returns a list of results — it's a more "functional programming" approach. A `for` loop does the same thing but requires manually initializing and accumulating results. `lapply()` is more concise and idiomatic in R. However, `for` loops can be easier to read and debug, especially when you need to handle errors or track progress. For API work with potential failures, a `for` loop with `tryCatch` or `warning()` can be more transparent.
:::

## Part 5: Reflection & Pitfalls

### Common Pitfalls When Working with APIs

1. **Rate Limits:** Most APIs limit how many requests you can make per minute/hour. The free OpenWeather tier allows 60 calls/minute. Design your code to respect these limits.

2. **Authentication Expiry:** API keys can expire or be revoked. Always check your key status if you get `401` errors.

3. **Data Structure Changes:** API providers can change their response format without warning. Always inspect the structure of new responses before building pipelines around them.

4. **URL Encoding:** Special characters (spaces, ampersands, accents) must be encoded. Use `URLencode()`.

5. **Nested JSON Complexity:** Real-world JSON can be deeply nested. Practice with `str()` and `names()` to navigate the structure before trying to flatten it.
